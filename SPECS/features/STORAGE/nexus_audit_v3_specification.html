Section 01

The Problem — Life Without Nexus Audit V3

Consider two analogies that perfectly describe the current state of developer tooling.

The School With No Unified Report Card

Imagine a school where each teacher keeps their own grade book in a completely different format. Mrs. Smith uses handwritten ledgers, Mr. Jones uses a private spreadsheet, and Ms. Williams uses a proprietary app with no export. At the end of term, the headteacher must phone each teacher individually to compile a final report. There is no early warning when a student is failing across multiple subjects simultaneously. There is no way to see whether performance has improved or declined over the past three terms. Every decision is made in the dark.

This is precisely the situation every Python developer faces with code quality tooling today.

The Airport With Five Separate Security Desks

Now imagine an airport where security is split across five independent desks, each checking a different thing, each with its own queue, its own output format, and its own record-keeping system. To board your flight you must visit all five desks, collect five different slips of paper, and then manually reconcile them yourself. If one desk was flagging more passengers than usual last week, there is no way to know — each desk's records are isolated.

Below is the technical equivalent of that airport: the current Python developer toolchain.

Fig 1 — The current state: four tools, four output formats, no shared history. The developer is the only connection between them.
The Real Cost

Studies of developer tool context-switching estimate 15–25 minutes of lost focus per unplanned tool switch. Running four separate audit tools after each significant commit costs an estimated 2–4 hours per developer per week in aggregate — not counting the time to manually reconcile findings and build trend reports.

The Specific Gaps That Nexus Audit V3 Closes

  • No unified score. Each tool has its own grading scale. There is no single number that answers "is this codebase getting better or worse?"
  • No trend data. Running Bandit today and again next week gives you two snapshots with no automatic diff, no persistence, no history.
  • No fix queue. When 47 findings exist, there is no ranked, state-tracked list of what to fix first. Every run starts from zero.
  • No coupling analysis. No existing lightweight tool maps inter-module dependencies and flags architectural drift as a first-class metric.
  • Fragmented CI integration. Each tool needs its own CI step, its own artifact upload, its own threshold configuration.

Section 04

Storage Architecture Specification

Directory Structure

~/.nexus_audit/
├── projects/
│   └── <project_id>/           # SHA-256 of the project root path
│       ├── nexus_state.db       # SQLite index — queryable, rebuildable
│       └── jobs/
│           └── <job_id>/       # ISO-8601 timestamp + random suffix
│               ├── audit_data_complete.json   # Full payload — source of truth
│               └── audit_summary.json         # Lightweight — for fast trend reads
└── config.json                  # Global configuration
Immutability Guarantee

Once written, job directories are never modified. Every run produces a new directory. The retention policy deletes the oldest directories when max_jobs is exceeded — it never edits existing ones. This makes the archive trustworthy as an audit log.

SQLite Index — Schema & WAL Configuration

The SQLite file at ~/.nexus_audit/projects/<id>/nexus_state.db is a queryable index over the JSON archives. It is always rebuildable; it is never the source of truth.

Mandatory initialization pragmas (run on every DB open)

PRAGMA journal_mode = WAL;        -- Concurrent reads during writes
PRAGMA synchronous = NORMAL;      -- Reduce fsync wait without losing durability
PRAGMA busy_timeout = 5000;       -- Wait 5s for locks instead of crashing
PRAGMA foreign_keys = ON;
PRAGMA cache_size = -8000;        -- 8MB cache

Schema

CREATE TABLE IF NOT EXISTS runs (
  run_id          TEXT    PRIMARY KEY,
  project_id      TEXT    NOT NULL,
  timestamp       INTEGER NOT NULL,          -- Unix epoch seconds
  job_dir         TEXT    NOT NULL,          -- Pointer back to JSON files
  git_commit      TEXT,
  git_branch      TEXT,
  git_author      TEXT,
  score_overall   REAL,
  score_security  REAL,
  score_quality   REAL,
  score_complexityREAL,
  score_deps      REAL,
  count_total     INTEGER,
  count_critical  INTEGER,
  count_high      INTEGER,
  duration_ms     INTEGER
);

CREATE TABLE IF NOT EXISTS findings (
  fingerprint     TEXT    NOT NULL,
  run_id          TEXT    REFERENCES runs(run_id),
  category        TEXT,                         -- security | quality | complexity | deps
  severity        TEXT,                         -- critical | high | medium | low
  rule_id         TEXT,
  file_path       TEXT,
  git_pointer     TEXT,                         -- JSON: {commit, lines:[start,end]}
  first_seen      TEXT    REFERENCES runs(run_id),
  status          TEXT    DEFAULT 'open',       -- open | suppressed | resolved
  suppressed_untilINTEGER                        -- Unix epoch: snooze expiry
);

CREATE INDEX IF NOT EXISTS idx_runs_project   ON runs(project_id, timestamp DESC);
CREATE INDEX IF NOT EXISTS idx_findings_fp    ON findings(fingerprint);
CREATE INDEX IF NOT EXISTS idx_findings_run   ON findings(run_id);
CREATE INDEX IF NOT EXISTS idx_findings_file  ON findings(file_path);
Why Git Pointers, Not Code Snippets

The git_pointer column stores {"commit":"a3f9c2b","lines":[42,48]} rather than copying source code into the database. This means the dashboard always renders the live file — not a stale snapshot from scan time — and keeps the database size proportional to findings count, not codebase size.

CLI Specification

All commands follow the pattern nexus <command> [target] [flags]. The tool is usable without any flags — sensible defaults apply everywhere.

CommandDescriptionKey Flags
nexus audit [path] Run a full audit. Writes JSON archive and upserts SQLite index. --watch live mode · --fast summary only · --json stdout
nexus diff <run_a> <run_b> Structural diff between two audit runs. Shows score delta, new/resolved findings, coupling changes. --format sarif|json|text · --severity HIGH+
nexus trend [--last N] Score trend for the last N runs. Reads from SQLite index in O(log N). --since 30d · --branch main · --csv
nexus queue Show the ranked fix queue: open findings ordered by severity × age × recurrence. --snooze <fp> 30d · --resolve <fp> · --assign <fp> alice
nexus export Export audit history in interoperable formats. --format sarif|json|csv · --since 90d · --runs 50
nexus rebuild-index Replay all JSON archives in the current project and regenerate nexus_state.db from scratch. --dry-run · --verbose
nexus clean Remove old job directories per retention policy. --keep 100 · --dry-run
nexus serve Start the local web dashboard. --port 7331 · --open

Exit Codes

0  Clean — score above threshold, no new CRITICAL/HIGH findings
1  Warning — new findings exist below threshold
2  Fail — new CRITICAL findings or score dropped below configured minimum
3  Error — tool failure (permissions, parse error, etc.)
Section 06

Killer Features

These three capabilities differentiate Nexus Audit V3 from every existing tool. They are only possible because of the local-first JSON archive design.

⚡

nexus diff — Structural Audit Diffs

Compare any two audit runs to produce a structured changelog: which findings are new, which were resolved, which scores improved or regressed, and which inter-module coupling edges appeared or disappeared. Answers the question every developer has after a score drop: "what specifically changed, and in whose commit?" No other local-first tool does this today.

🔄

rebuild-index — Zero-Risk Database

If the SQLite index is ever deleted, corrupted, or migrated to a new schema, a single command replays every JSON archive and reconstructs it from scratch. This makes SQLite a read-optimized performance cache — never a single point of failure. JSON files are always the source of truth. Developers who are cautious about databases can rely on this guarantee.

🔗

SARIF Export — Universal Compatibility

SARIF (Static Analysis Results Interchange Format) is the OASIS standard consumed natively by GitHub Code Scanning, VS Code's Problems panel, GitLab's Security Dashboard, and Azure DevOps. One command exports Nexus Audit findings in SARIF so they appear inline in pull requests, in the editor gutter, and in enterprise security dashboards — with zero additional integration work.

Section 07

Implementation Guide

This section is written for developers who are new to the codebase, or who are cautious about the storage decisions. Every step explains why, not just what. Take it one phase at a time — each phase is independently shippable.

If you are feeling uncertain

Start with Phase 1 only. JSON-only storage is a fully working product. Phases 2–4 are additive improvements you can ship when you are ready. Nothing in Phase 2 breaks Phase 1. The JSON files you write in Phase 1 will still be readable in Phase 4.

Phase 1 — JSON-Only Storage (Ship This First)

Create the project directory structure

On first run, detect the project root (walk up from CWD until you find a .git directory or reach /). Compute the project ID as sha256(str(project_root))[:16]. Create ~/.nexus_audit/projects/<id>/jobs/. This never fails — it is just Path.mkdir(parents=True, exist_ok=True).

import hashlib, os
from pathlib import Path

def project_dir(project_root: Path) -> Path:
    pid = hashlib.sha256(str(project_root).encode()).hexdigest()[:16]
    d = Path.home() / ".nexus_audit" / "projects" / pid
    d.mkdir(parents=True, exist_ok=True)
    return d
Directory created, no errors thrown on a clean machine.

Generate a job ID and write the JSON archive

Use ISO-8601 timestamp + git short hash for the job ID — human-readable and sortable. Write audit_data_complete.json atomically: write to a .tmp file first, then os.replace() to the final path. This prevents a corrupted file if the process is killed mid-write.

import json, time
from datetime import datetime, timezone

def write_audit(project_dir: Path, payload: dict) -> Path:
    ts = datetime.now(timezone.utc).strftime("%Y-%m-%dT%H-%M-%SZ")
    commit = payload.get("git", {}).get("commit", "unknown")[:7]
    job_id = f"{ts}_{commit}"
    job_dir = project_dir / "jobs" / job_id
    job_dir.mkdir(parents=True)

    # Atomic write — never leaves a half-written file
    tmp = job_dir / "audit_data_complete.json.tmp"
    tmp.write_text(json.dumps(payload, indent=2))
    os.replace(tmp, job_dir / "audit_data_complete.json")

    # Write lightweight summary for fast trend reads
    summary = {k: payload[k] for k in ["scores", "counts", "git", "timestamp"]}
    summary["finding_fingerprints"] = [f["fingerprint"] for f in payload["findings"]]
    (job_dir / "audit_summary.json").write_text(json.dumps(summary, indent=2))
    return job_dir
After running an audit, two JSON files exist in the job directory. cat them — they should be human-readable.

Implement trend queries using summary files

For Phase 1, trend queries read and parse the summary files in a directory listing. This is O(N) — sufficient for up to ~150 runs. Sort job directories by name (ISO timestamp prefix means alphabetical = chronological).

def load_trends(project_dir: Path, limit: int = 50) -> list[dict]:
    jobs_dir = project_dir / "jobs"
    job_dirs = sorted(jobs_dir.iterdir(), key=lambda p: p.name, reverse=True)[:limit]
    results = []
    for job_dir in reversed(job_dirs):
        summary_path = job_dir / "audit_summary.json"
        if summary_path.exists():
            results.append(json.loads(summary_path.read_text()))
    return results
Phase 1 is complete and shippable. Stop here if you want to validate the product before adding more complexity.

Phase 2 — SQLite Index

Open the database with production pragmas

Always run these pragmas immediately after connect(). They take microseconds but prevent entire classes of concurrency and durability bugs. Write this as a context manager so you cannot forget.

import sqlite3
from contextlib import contextmanager

@contextmanager
def db(project_dir: Path):
    path = project_dir / "nexus_state.db"
    conn = sqlite3.connect(str(path))
    conn.row_factory = sqlite3.Row
    conn.executescript("""
        PRAGMA journal_mode = WAL;
        PRAGMA synchronous = NORMAL;
        PRAGMA busy_timeout = 5000;
        PRAGMA foreign_keys = ON;
        PRAGMA cache_size = -8000;
    """)
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise
    finally:
        conn.close()

Implement rebuild-index before you write the upsert

Write rebuild-index first. This validates that your schema is correct before any real data goes in, and gives you the safety net that makes all subsequent work less risky. It is also the correct mental model: SQLite is always derivative of JSON.

def rebuild_index(project_dir: Path):
    """Replay all JSON archives → rebuild SQLite from scratch."""
    db_path = project_dir / "nexus_state.db"
    if db_path.exists():
        db_path.rename(db_path.with_suffix(".db.bak"))  # Safety copy

    with db(project_dir) as conn:
        _create_schema(conn)
        jobs_dir = project_dir / "jobs"
        for job_dir in sorted(jobs_dir.iterdir()):
            summary = job_dir / "audit_summary.json"
            if summary.exists():
                _upsert_run(conn, json.loads(summary.read_text()), job_dir)

    db_path.with_suffix(".db.bak").unlink(missing_ok=True)  # Clean up
    print(f"Index rebuilt from {len(list(jobs_dir.iterdir()))} runs.")
Run rebuild-index on a project with existing JSON files. The SQLite file should be created. Run it twice — the second run should produce the same file.

Add the upsert call to the audit pipeline

After the JSON files are written (Phase 1 code, unchanged), call the upsert function. This is the only code change to the audit path in Phase 2 — everything else is additive.

def _upsert_run(conn, summary: dict, job_dir: Path):
    conn.execute("""
        INSERT OR REPLACE INTO runs
        (run_id, project_id, timestamp, job_dir,
         git_commit, git_branch, git_author,
         score_overall, score_security, score_quality,
         score_complexity, score_deps,
         count_total, count_critical, count_high, duration_ms)
        VALUES (?,?,?,?,?,?,?,?,?,?,?,?,?,?,?,?)
    """, (
        summary["job_id"],
        summary["project_id"],
        int(datetime.fromisoformat(summary["timestamp"]).timestamp()),
        str(job_dir),
        summary.get("git", {}).get("commit"),
        summary.get("git", {}).get("branch"),
        summary.get("git", {}).get("author"),
        summary["scores"]["overall"],
        summary["scores"]["security"],
        summary["scores"]["quality"],
        summary["scores"]["complexity"],
        summary["scores"]["dependencies"],
        summary["counts"]["total"],
        summary["counts"]["critical"],
        summary["counts"]["high"],
        summary.get("duration_ms")
    ))
After a new audit run, query the SQLite file with any SQLite browser. The run should appear in the runs table.

Phase 3 — nexus diff

Load both JSON archives and diff the finding sets

The fingerprints in each audit_summary.json are enough to compute which findings are new and which were resolved. Load the full payload from audit_data_complete.json only for findings that need detail.

def diff_runs(job_dir_a: Path, job_dir_b: Path) -> dict:
    a = json.loads((job_dir_a / "audit_summary.json").read_text())
    b = json.loads((job_dir_b / "audit_summary.json").read_text())

    fps_a = set(a["finding_fingerprints"])
    fps_b = set(b["finding_fingerprints"])

    new_fps     = fps_b - fps_a      # In B but not A — new findings
    resolved_fps = fps_a - fps_b     # In A but not B — resolved
    persisted_fps = fps_a & fps_b    # In both — unchanged

    score_delta = {k: b["scores"][k] - a["scores"][k] for k in b["scores"]}

    return {
        "score_delta": score_delta,
        "new_count": len(new_fps),
        "resolved_count": len(resolved_fps),
        "persisted_count": len(persisted_fps),
        "new_fingerprints": list(new_fps),
        "resolved_fingerprints": list(resolved_fps),
    }

Phase 4 — SARIF Export

SARIF 2.1.0 — Minimum Required Fields

SARIF is a JSON format. The minimum valid SARIF document needs: version, $schema, runs[].tool.driver (with name and rules), and runs[].results (each with ruleId, message.text, and locations). That is it. GitHub and VS Code accept any document that satisfies this minimum.

def to_sarif(findings: list[dict]) -> dict:
    return {
        "version": "2.1.0",
        "$schema": "https://raw.githubusercontent.com/oasis-tcs/sarif-spec/master/Schemata/sarif-schema-2.1.0.json",
        "runs": [{
            "tool": {"driver": {"name": "Nexus Audit V3", "version": "3.0.0", "rules": []}},
            "results": [{
                "ruleId": f["rule_id"],
                "level": {"critical": "error", "high": "error", "medium": "warning", "low": "note"}.get(f["severity"], "warning"),
                "message": {"text": f["message"]},
                "locations": [{"physicalLocation": {"artifactLocation": {"uri": f["file_path"]}}}]
            } for f in findings]
        }]
    }

Write Strategy: In-Memory Buffer with Debounced Flush

For watch mode (nexus audit --watch), Nexus triggers a scan on file save. Writing the full JSON payload and updating SQLite on every save is wasteful. The correct pattern:

  1. Scan completes → update in-memory state immediately. The dashboard reflects changes in microseconds, no disk I/O needed.
  2. Debounce for 3–5 seconds of file-save silence, then write the JSON archive and SQLite upsert as a single sequential operation.
  3. Multiple rapid saves in the debounce window become one write. This dramatically reduces disk writes without any perceptible dashboard lag.
Section 09

Appendix — Decisions & Rationale

This section documents every significant architectural choice with the reasoning behind it, so future contributors understand not just what was built but why alternatives were rejected.

Decision Log

DecisionAlternatives RejectedReason
JSON as source of truth SQLite only; PostgreSQL Human-readable, zero-dependency, portable. A developer should be able to inspect any audit result with a text editor, 10 years from now, without any special software.
SQLite as queryable index Re-parsing summaries; DuckDB; Redis SQLite is a C library bundled in Python's stdlib. It requires zero setup, handles O(log N) indexed queries, and is already the precedent set by Grype for exactly this use case.
Per-project SQLite file Single global nexus_core.db A per-project file travels with the project when zipped or shared. A global file is a new form of lock-in that undermines the portability value proposition.
No PostgreSQL PostgreSQL + Elasticsearch (SonarQube model) Provisioning PostgreSQL takes minutes to days and requires a running server process. This is the exact setup friction we are eliminating. There is no query complexity that warrants it at local-developer scale.
Git pointer references, not code snippets Storing source code in the payload Copying code into the database causes payload size to grow with codebase size, not findings count. Pointers are always accurate (they reference the live file) and keep payloads proportional.
SARIF output over custom format Proprietary export formats SARIF is the OASIS standard. GitHub, VS Code, GitLab, and Azure DevOps all consume it natively. Using it costs nothing and unlocks the entire developer toolchain ecosystem at launch.
Immutable job directories Overwriting the last run's files Immutability makes the archive trustworthy as an audit log. History is never lost. The retention policy only deletes the oldest jobs when the cap is hit — it never edits.
WAL + NORMAL + busy_timeout pragmas Default SQLite settings Default SQLite settings (DELETE journal mode, FULL synchronous) cause write lock contention in concurrent scenarios and unnecessary fsync overhead. The three pragmas are the minimal production-grade configuration.
